# 🛡️ ForTem ↔ 인게임 연동 및 통합 보안/아키텍처 점검 체크리스트 (Master)

이 문서는 과거 개발 페이즈(Phase 1~4)를 거치며 발견되었던 **P0(치명적 결함)부터 P3(아키텍처 부채)까지의 모든 인게임-블록체인 연동 이슈**를 총망라한 마스터 체크리스트입니다. Phase 5 코어 개발 및 향후 유지보수를 진행할 때, 본 문서의 항목들이 코드베이스 방어선에 완벽히 구축되어 있는지 항시 교차 검증해야 합니다.

---

## 🚨 1. P0 Class: 치명적 사고 및 자산 탈취 방어 (Critical Security & Assets)

- [x] **[타인 자산 탈취 방어] API 소유권 통제 (Broken Access Control)**
  - **증상**: 프론트에서 넘어오는 `{"game_user_id": "user_A"}`를 백엔드가 맹신하면, 해커가 패킷을 조작해 user_B의 자산을 강제 민팅/탈취 가능.
  - **방어 로직**: 모든 엔드포인트(`mint/export`, `inventory/batch`)에서 `supabase.auth.getUser(token)` 미들웨어 검증을 강제하고, 세션 유저와 조작 대상 ID가 일치하는지 교차 검증해야 함.
- [x] **[블록체인 생태계 붕괴] NFT 중복 민팅 (Double-Spending) 원천 차단**
  - **증상**: 유저가 의도적으로 '민트(Export)' 버튼을 다다닥 연타(Race Condition) 시, SELECT와 UPDATE 쿼리 틈새를 파고들어 1개의 아이템이 2~3개의 NFT로 중복 발행됨.
  - **방어 로직**: SELECT 후 UPDATE가 아닌, `UPDATE ... WHERE is_minted = false` 형태의 단일 원자적(Atomic) 잠금 쿼리로 동시성 요청을 물리적으로 튕겨내야 함.
- [x] **[아이템 무한 복사] 웹훅 멱등성(Idempotency) 및 고유 식별자(UID) 충돌 보호**
  - **증상**: 웹훅(Webhook)이 네트워크 지연으로 2~3회 재전송될 경우, 동일한 콜백에 대해 아이템 상태 갱신이나 지급이 중복 처리될 수 있음.
  - **방어 로직**: DB의 `item_uid` 및 `redeem_tx_hash` 컬럼에 `UNIQUE` 제약 조건을 걸어, 동일한 트랜잭션 해시나 아이템이 물리적으로 두 번 INSERT/UPDATE 되는 것을 DB 엔진 단에서 거부해야 함.

## ⚠️ 2. P1 Class: 유저 경험 파괴 및 시스템 결함 (UX & System Failures)

- [x] **[자산 증발] 민팅 실패 시 영구 잠김 (Rollback Failure)**
  - **증상**: 가스비 부족이나 ForTem 네트워크 다운 시 발생한 에러를 잡지 못하면, 인게임 DB에서 아이템이 영원히 `minting_in_progress` 상태로 갇혀버려 유저 자산이 '증발'함.
  - **방어 로직**: 실패 Catch 블록 시 즉각 `status: 'mint_failed'`, `is_minted: false`로 복구(Rollback)하고, 클라이언트 UI에 '재시도(Retry)' 버튼을 활성화시켜야 함.
- [x] **[정보 유출 및 조작] 서버 환경변수와 에러 객체 격리**
  - **증상**: `network: "sui-mainnet"`이나 `studio_id`를 클라이언트 페이로드로 받으면 해커가 타겟 네트워크를 임의 조작 가능. 클라이언트로 전송되는 500 에러 스택에 DB 테이블명 등 백엔드 정보 노출.
  - **방어 로직**: 중요 키값은 서버 `process.env`에서만 관리. 에러 응답은 "Minting failed. Pleas try again" 등 마스킹 처리된 범용 메시지만 반환.
- [x] **[오입금 소실] 지갑 주소 형식 하드 검증**
  - **증상**: 실수로 아무 문자열이나 지갑 주소로 입력했는데 그대로 민팅 통과 시 자산 영구 미아 발생.
  - **방어 로직**: 클라이언트 폼(Form)과 백엔드 최종 검증 양쪽에서 Sui 주소 정규식(`/^0x[a-fA-F0-9]{64}$/`) 검증 루틴 가동.

## 🛠️ 3. P2 Class: 인게임 어뷰징 통제 및 동기화 무결성 (Anti-Abuse & Sync)

- [x] **[먹튀 방지] 인게임 지연 저장(Batch Save) 아키텍처**
  - **증상**: 게임 중 획득한 아이템이 즉시 DB에 저장되면 희귀템만 먹고 고의 강제 종료(ALT+F4)하는 체리피킹 어뷰징 발생.
  - **방어 로직**: 획득템은 메모리(`pendingInventory`)에만 쌓아두고, **1) 보스방 클리어 2) 플레이어 사망** 시에만 백엔드에 `Batch Insert` 되도록 세이브 시점 통제.
- [x] **[지갑 유실 방지] 브라우저 종속성 탈피**
  - **증상**: 연동된 지갑 주소를 `localStorage`에만 저장 시, 폰/PC 기기 변경 시 매번 지갑을 다시 입력해야 하며 로컬 데이터 날아가면 게임 진행 차질.
  - **방어 로직**: Supabase `profiles` 테이블 등에 지갑 주소를 영속화하여 어떤 기기로 접속해도 내 지갑이 연동되도록 관리.
- [x] **[스탯 위변조] 메타데이터 임의 주입 방어**
  - **증상**: 클라이언트가 보내는 `metadata` 객체를 별도 필터링 없이 ForTem API에 바로 꽂으면, 유저가 브라우저 콘솔을 열고 Attack을 9999로 변조해 NFT를 발행할 수 있음.
  - **방어 로직**: 서버단에서 넘어온 메타데이터 중 `['Attack', 'Defense', 'HP', 'Speed']`와 같이 허가된 화이트리스트 속성만 안전하게 추출(Sanitize)하여 민팅 API에 전달.

## 🧱 4. P3 Class: 구조적 부채 (Architecture Debt)

- [x] **클라이언트 로드 전 Race Condition 픽업**
  - **설명**: 게임 씬 로딩(`GameCanvas`)과 `getUser()` 비동기 유저 정보 획득 속도 차이로 인해, 유저 정보가 아직 로드되지 않은 상태에서 초반 1~2초간 몬스터의 드랍 아이템을 먹으면 `userId=null`이라 아이템이 미저장 처리(Silent miss) 됨.
  - **개선**: 유저 세션 로딩이 완전히 끝난 후(`isLoading: false`)에만 게임 씬 입장이 가능하도록 Splash/Loading Screen 방어 타설.
- [x] **웹훅 타이밍 공격(Timing Attack) 및 Replay 공격 방어**
  - **설명**: 웹훅 서명 검증 시 `===` 비교는 바이트 분석에 취약하므로 `crypto.timingSafeEqual` 도입 시급. 또한 가로챈 과거 웹훅 페이로드를 다시 쏘는 Replay 공격 방지를 위해 타임스탬프 허용 오차폭(예: 5분 이내) 검증 도입 필요.
