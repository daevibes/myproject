## 원본 체크리스트의 함정 3가지

### 함정 1. 에셋(Assets)을 1단계에 넣으면 개발이 막힌다

"캐릭터, 배경, 아이템 이미지 파일 배치"는 **PO가 먼저 결정해야 할 기획 사항**입니다.
에셋이 없어도 개발은 시작할 수 있습니다. 임시로 색깔 사각형(placeholder)으로 대체하고 게임 로직을 먼저 완성하는 것이 업계 표준 방식입니다.

개발 착수 전 PO가 결정해야 할 항목:

| 결정 항목 | 왜 지금 필요한가 |
|----------|----------------|
| 게임 장르/컨셉 | 어떤 에셋을 구해야 할지 모름 |
| 캐릭터 종류 수 | Sprite Sheet 구조가 달라짐 |
| **아이템 이름 목록** | DB의 `in_game_item_id`와 반드시 일치해야 함 |
| 캔버스 해상도 | 에셋 해상도를 맞춰야 함 |

---

### 함정 2. 4단계(백엔드 연결)는 3단계(드롭 시스템) 없이 테스트 불가

체크리스트에서 4단계가 "중요"로 표시됐지만, **3단계 드롭 시스템이 4단계의 선행 조건**입니다.

```
아이템 드롭 (3단계) ──→ 아이템 획득 이벤트 (4단계) ──→ API 전송
```

또한 현재 인벤토리 API의 `in_game_item_id` 형식(`sword_lv1` ~ `sword_lv99` 랜덤)과 **게임에서 드롭되는 아이템 ID가 일치해야 합니다.** PO가 아이템 ID 목록을 사전에 확정해서 개발자에게 전달해야 합니다.

---

### 함정 3. 5단계 "반응형 대응"은 1단계에서 함께 설계해야 한다

Phaser는 기본적으로 고정 캔버스 크기로 동작합니다. 반응형 대응을 나중에 추가하면 게임 전체 좌표계를 다시 잡아야 할 수 있습니다. **캔버스 크기 결정 시 Scale Manager 설정을 함께 잡아야 합니다.**

---

## 권장 착수 순서 (스프린트 재구성)

원본 체크리스트 순서 대신 아래 4개 스프린트로 진행을 권장합니다.

---

### Sprint 0 — PO 결정 사항 (개발 착수 전)

개발자에게 작업을 넘기기 전에 PO가 먼저 결정해야 합니다. 이것이 없으면 개발자가 중간에 막힙니다.

```
① 아이템 ID 목록 확정
   예시: sword_fire, sword_ice, shield_iron, potion_hp
   → DB의 in_game_item_id 값과 100% 일치해야 함

② 아이템별 드롭 확률 결정
   예시: sword_fire 5%, potion_hp 30%

③ 게임 캔버스 크기 결정
   예시: 800x600 고정 or 전체 화면 비율형(FIT)

④ 에셋 방향 결정
   (A) 무료 에셋 사용 (kenney.nl 등)
   (B) 디자이너에게 의뢰
   (C) 임시 placeholder로 로직 먼저 완성 ← 단기 권장
```

---

### Sprint 1 — 게임 화면 뜨는 것 확인 (예상 1~2일)

```
① app/game/page.tsx 생성 (Phaser 캔버스 마운트)
② Phaser 기본 Scene 구성 (배경색만 있어도 OK)
③ 캔버스 크기 + Scale Manager 설정 (반응형 기초 포함)
④ /game 페이지 접속 시 빈 게임 화면 확인
```

**완료 기준:** 브라우저에서 `/game`을 열면 게임 화면(빈 캔버스)이 뜬다.

---

### Sprint 2 — 게임 로직 완성 (예상 3~5일)

```
① 캐릭터 이동 (색깔 사각형 placeholder로 시작)
② 몬스터/상자 배치 + 충돌 감지
③ 드롭 시스템 (Sprint 0에서 확정된 아이템 ID 사용)
④ HP, Score 상태 변수 (zustand 활용)
```

**완료 기준:** 캐릭터가 움직이고, 상자를 부수면 아이템이 드롭된다.

---

### Sprint 3 — 백엔드 연결 (예상 1~2일)

```
① 아이템 획득 시 POST /api/inventory 호출
   body: { game_user_id: "usr_test_12345", in_game_item_id: "확정된 아이템 ID" }
② 성공 시 게임 화면에 "획득 완료!" 알림 표시
③ in_game_item_id 일치 여부 확인
④ 게임 화면 내 /inventory 페이지 이동 버튼 추가
```

**완료 기준:** 게임에서 아이템을 먹으면 Supabase DB에 저장되고, `/inventory`에서 확인된다.

---

### Sprint 4 — 에셋 교체 + 마무리 (별도 일정)

```
① placeholder → 실제 이미지로 교체
② 로딩 화면 (프로그레스 바)
③ 최종 반응형 조정
```

**완료 기준:** 실제 게임 에셋으로 플레이 가능한 상태.

---

## 개발자 전달 시 반드시 포함할 정보

체크리스트와 함께 아래 4가지를 반드시 같이 전달해야 합니다:

```
1. 아이템 ID 목록 (Sprint 0 결과물)
   → 개발자가 드롭 시스템 코딩 시 사용할 기준값
   → 예: ['sword_fire', 'sword_ice', 'shield_iron', 'potion_hp']

2. 아이템별 드롭 확률표
   → 예: sword_fire 5%, shield_iron 10%, potion_hp 30%

3. 캔버스 크기 결정값
   → 예: 800x600, Scale Mode: Phaser.Scale.FIT

4. 현재 API 연동 스펙 (이미 구현된 것)
   → POST /api/inventory
   → body: { game_user_id: string, in_game_item_id: string }
   → game_user_id 현재값: "usr_test_12345" (테스트용)
```
