# Phase 4 업데이트 내역 요약 (ForTem NFT & 백엔드 연동)

이 문서는 게임 내 아이템의 백엔드 영속성(Persistence) 확보 및 ForTem 컬렉션으로의 NFT 내보내기 기능 통합(Phase 4) 과정에서 수정·개선된 사항들을 정리한 요약본입니다.

## 1. 🗄️ 백엔드 및 클라이언트 상태 연동 (핵심 구현)
- **GameCanvas 하이드레이션**: 게임 마운트 시 `GameCanvas`에서 Supabase 세션의 `user.id`를 가져와 Zustand Store(`useGameStore`)에 즉시 적재(Hydrate)하여 `game_user_id` 인증 및 식별 흐름 확보.
- **낙관적 업데이트(Optimistic Update) 패턴**: 몬스터 사냥 후 아이템 획득 시 로컬 Zustand 스토어에 임시 보관(`pendingInventory`)하고 실시간 API 호출은 차단(먹튀 어뷰징 방지).
- **게임 종료 시 일괄 저장(Batch Save)**: 몬스터 피격(Game Over)이나 스테이지 클리어, 또는 사용자의 합법적인 중도 포기(ESC 로비 나가기) 시에만 `POST /api/inventory/batch` API를 호출하여 아이템 데이터를 DB에 일괄 영속화.
- **`item_uid` 고유 락(Lock) 제약**: 다중 묶음 민팅 시 발생할 수 있는 PGRST116(`.single()`) 에러와 중복 처리를 방지하기 위해 `item_uid` 단일 발급 기반 잠금 적용 완료.

## 2. 🔐 지갑 연동 및 민팅 프로세스 최적화
- **Sui 지갑 필수 요건 제거**: 번거로운 `@mysten/dapp-kit` 종속성을 덜어내고, ForTem 지갑 주소를 유저가 수동으로 1회 입력·저장하는 직관적인 UX로 변경.
- **수정 불가 및 확인 절차**: 지갑 주소 오기입 방지를 위해 1회 입력 후 수정을 불가하게 잠그기 전 강한 확인(Confirm) 절차 도입.
- **아이템 소각(Burn) 안내 강화**: NFT로 내보내어 블록체인 상으로 아이템을 민팅할 때, 게임 내부 인벤토리에서는 해당 아이템이 영구 삭제(소각)됨을 알리는 `Confirm` 팝업을 추가하여 민팅 후 발생할 수 있는 유저 혼선 방지.

## 3. 🎨 인벤토리 UX/UI 전면 개편 (분리 및 고도화)
이전에는 게임 내 장착용 템과 NFT 후보 템이 하나의 화면에 뒤섞여 발생하던 혼동을 해결하고자 설계사상을 전면 수정했습니다.

* **인게임 로비 인벤토리와 NFT 상점의 완전한 분리**
  - **로비(사냥터) HUD**: 게임 내 장착/소각 등 기본적인 인게임 상호작용에만 집중. (나중에 모든 템이 NFT화될 것을 대비해 불필요하게 가시성을 해치는 ★표시 등은 제거 완료)
  - **NFT 상점 페이지 (`/inventory`)**: NFT 추출 전용 "분리된 관심사"를 가진 페이지로 성격 재정의.

* **NFT 추출 페이지(`/inventory`) Flow 리뉴얼**
  1. **가치 있는 아이템만 노출**: 일반 장비(Common, Rare) 여부 등 희귀성(Rarity)과 관계없이, 추후 ForTem 대시보드에 컬렉션으로 등록하고 `isExportable: true` 속성을 부여받은 민팅 대상(💎NFT 뱃지 보유) 아이템들만 격리하여 진열.
  2. **혼동 유발 액션 제거**: NFT 상점 내에서의 '게임 장착/소각' 기능과 장착 중인 상태 UI를 모조리 제거. 현재 게임 내에서 "장착 중"인 아이템은 내보내기가 불가능하도록 비활성화 처리.
  3. **[글로벌 1건 내보내기] 도입**: 
     - 각 아이템 카드마다 달려 있던 개별 Export 버튼을 제거하여 UI 복잡도를 덜어냄.
     - 대신, 유저가 민팅할 아이템 **딱 1개**를 클릭(선택)하면, 화면 하단에 고정된 거대한 **[💎 선택한 아이템 내보내기 (Export)]** 버튼이 활성화되는 플로우로 전환.
     - 로딩(민팅 중) 상태일 때는 여러 아이템을 동시다발적으로 클릭하여 어뷰징하거나 에러를 유발하는 다중 요청(Simultaneous Requests)을 방지(`disabled=true`).

## 4. 🛠️ SDK 패치 및 부채 해결
- **ForTem SDK 최신화 적용**: 최신 SDK 스펙에 맞춰 코드베이스 내 `secretKey` 매개변수를 `apiKey`로, `mint.exportItem` 메서드 호출을 `items.create`로 일괄 수정. (향후 SDK 메이저 업그레이드 시 참고 이력)

## 향후 과제 (Phase 5 진입 전 주요 보완점)
- **인벤토리 재하이드레이션 (DB → Zustand) 동기화 미정의**: 현재 Zustand 상태는 `localStorage` persist에만 의존하므로, 다른 기기 접속이나 쿠키 초기화 시 인벤토리 데이터가 소실됨. 게임 시작 시 `GET /api/inventory` 엔드포인트를 호출하여 기 데이터베이스와 동기화하는 로직 추가 필요.
- **`mint_failed` 상태 UX 미정의**: DB에 아이템 민팅 실패 상태가 기록되더라도, 유저에게 재시도를 유도하는 CTA(Call To Action)나 실패 안내 팝업 등의 UI 처리가 누락됨.
- **ForTem Webhook 신규 구현**: 민팅 완료/실패에 따른 상태 반영용 Callback 처리를 위한 `/api/webhook/fortem` 웹훅 엔드포인트 자체가 미존재하며, 테스트 파이프라인 단계를 넘어 실제 신규 구현이 시급함.
